# OA Agent-Assisted Install — Maturity Model + One-Shot Install-Agent Design

**Companion to** `OA-SETUP-FEASIBILITY.md` (same branch). That doc answered *"could a fresh agent bootstrap OA today?"* (answer: not autonomously, and the published npm path is dead). This doc answers the **design** follow-on: **the best way for an agent to guide a user through a one-shot install to a proven, running, mission-advancing state** — including a maturity model, per-profile completion criteria, the install-agent flow, and the concrete build plan.

**Sources.** open-autonomy @ `research/oa-setup-feasibility` (rebased on `main` @ cefa626) **plus the five in-flight PRs #137–#141** (fetched as `pr-137`…`pr-141`), which are the machinery this design must build on. Two Fable‑5 diggers + a Fable‑5 skeptic pass. Citations are `file:line` (PR-branch cites are marked `#pr-NNN`). Backing evidence: `findings-E-prs-137-141.md`, `findings-F-maturity-signals.md`, and the four earlier `findings-*.md`.

**Post-merge baseline (what #137–#141 add).** #138 makes **direction docs first-class**: `documents.roles: {vision (required), constitution?, roadmap?}` is compile-validated, auto-gated into `human_required_paths`, and **hard-FAILs preflight if the file is missing** (`packages/core/src/ir.ts:86,93-96,137-163`#pr-138; `ir-yaml.ts:36-46`#pr-138; `scripts/open-autonomy-preflight.test.ts:135-140`#pr-138). #140 ships the **`@volter/oa` local-runner CLI** with a deterministic `oa doctor [--live] [--json]`, `oa status`, `oa start/once/pause/resume/dispatch` (`packages/local-runner-cli/src/index.ts:41-52`#pr-140) — **but no `oa init`/`oa up`**. #137 ships the **simple-gh `audit` agent** — a nine-check conformance auditor (`profiles/simple-gh/skills/audit/SKILL.md:64-144`#pr-137), prose-only, dispatch-only, simple-gh-only, aimed at *drift in a running install*. #139 ships **vision-anchored `planner`s** that self-replenish the board at *draft* altitude. #141 converges self-driving onto the sdlc core and finally ships it a `provision.json`. **None of the five is a setup agent, a profile recommender, or a maturity model** — verified by grep across all five branches (`findings-E`).

## Q1 — The Maturity Model

### Does OA have an explicit one? No.
**There is no explicit, named stage/readiness model in OA today** (verified: no "maturity"/"readiness"/"stage" artifact in `main` or any of #137–#141). The ladder is **implicit** — reconstructed from guards that each police exactly one transition (clobber guards → scaffolded; the uncommitted-harness guard → committed; the paused marker → running-authorized; `doctor` → proven). Nothing composes those signals, and **no artifact records which stage an install is at**. Worse, the two most operator-visible transitions — *unpause* and *arm auto-merge* — leave **no durable record**: unpause is a file deletion (`packages/substrate-local/src/emit.ts:106-122`), and the paused marker is *deliberately excluded from the manifest* so nothing can even ask "was this ever paused?" (`emit.ts:602-612`). **This absence is itself a finding the owner asked about.**

### Proposed: the OA Install Maturity Model (IMM) — M0…M6
One ladder, same stages for every profile; **the signals that mark each transition differ per profile** (§ per-profile matrix below).

**Framing (don't over-sell the model).** The *observable core is M2–M5* — and it maps almost 1:1 onto OA's **existing** `compile → preflight/doctor → unpause` chain. **M0** is vacuous, **M1** has no artifact today (it is defined into existence to name the profile/direction decision), and **M6** has no signal strong enough to prove on any profile yet. So the honest contribution is *not* a brand-new seven-rung taxonomy but: **(1) formalize the implicit chain OA already enforces via scattered guards, (2) compose its signals into a single stage verdict + a durable record (there is none today — even unpause and arm-auto-merge leave no trace, `emit.ts:602-612`), and (3) add the one genuinely missing rung — an "advancing" telemetry signal.** M0/M1 and M6-on-simple-* are explicitly *aspirational* markers, not things a validator can read today. With that caveat, the ladder:

| Stage | Name | Definition | Marking signal(s) (see `findings-F` A#) |
|---|---|---|---|
| **M0** | **EMPTY** | Repo exists, no OA. | No `.open-autonomy/`. |
| **M1** | **SCOPED** | A profile + substrate chosen, and the direction *intent* captured. | *No artifact today* — propose recording it (see build plan). Irreducibly needs a human decision unless a recommender picks it. |
| **M2** | **SCAFFOLDED** | `compile` succeeded; the harness exists on disk; the fence is seeded. | `.open-autonomy/generated.json` valid (`open-autonomy.generated.v1`, every `files[]` entry on disk) — A1/A2 (`packages/core/src/file-manifest.ts:14-35`; `bin/autonomy-compile.ts:265-266`); `autonomy.yml` parses — A3; `.open-autonomy/paused` seeded on fresh compile — A4 (`packages/substrate-local/src/emit.ts:480-489,508-517`). |
| **M3** | **INSTALLED** | Env/deps present, harness committed, plumbing proven; on GitHub, the repo is provisioned. | **Local:** `preflight` exit 0 — A11 (`bin/preflight.ts:882-917`); uncommitted-harness guard passes — A6 (`emit.ts:269-342`); `oa doctor` exit 0 (checks 1–6) — A7/A8/A10 (`bin/doctor-checks.ts:20-22,1171`). **GitHub:** `open-autonomy-preflight` `ready:true` — A12 (`scripts/open-autonomy-preflight.ts:196-202`); branch protection applied from `provision.json` — A13. |
| **M4** | **ARMED** | Direction docs present, filled, and gated; the board holds ≥1 actionable item. | **Direction:** `documents.roles.vision` declared + file exists (hard preflight FAIL otherwise) — pr-138 (`ir.ts:137-163`#pr-138); **+ [gap] content is not the `<!-- REPLACE THIS -->` template** (nothing checks this today). **Board:** a **deterministic** dispatchable-item predicate now ships — pr-140's `packages/local-runner-cli/src/eligibility.ts:15-55`#pr-140 (both `ztrack` ready-list and `gh-issues` ready-minus-parked variants, which the reconciler itself gates on). *Caveat:* it does **not** encode simple-sdlc's `oa-approved` allowlist (`profiles/simple-sdlc/ir.yml:80-83`) and its identity defaults (manager→ztrack, pm→gh-issues) would misfire for simple-sdlc's ztrack-board `pm` — so the M4-board check must build on `eligibility.ts` *plus* the profile's allowlist. |
| **M5** | **RUNNING** | The fence is lifted and a **profile agent actually fired** (not merely "the loop is up"). | `.open-autonomy/paused` absent — A5 (`emit.ts:106-122`); a real profile-agent session appears via `oa status` / `oa once` (A5). **Caution — what "dispatch-proven" really means:** there are **two** doctors and neither proves a *profile tick advanced*. The real-session check (`bin/doctor-checks.ts:1024-1136`, on **main**, invoked as `npx open-autonomy doctor --live`, i.e. **behind the dead npm path**) launches a **throwaway probe agent** (`oa-doctor-probe-<pid>`) with a synthetic prompt via `AUTONOMY_PROMPT_DIR`, **bypassing schedule/eligibility/skills**, and PASSes if the session merely survives ~30s — so it proves the *launch chain + CLI auth + provider attach*, not that the profile loop works. pr-140's `oa doctor --live` is weaker still: it only probes provider `/healthz` (`packages/local-runner-cli/src/doctor.ts:23-45`#pr-140). **This is "setup complete + running."** |
| **M6** | **ADVANCING** | A gate-passed, merged PR closed a mission-linked work item — the loop is *advancing the mission*, not merely ticking. | **No shipped signal is strong enough today — this rung is aspirational on every profile.** The nearest proxies are both *weaker* than the definition: `reconcile-merged-issues` closes an issue on **any** merged PR whose branch is `agent/issue-<N>` — a docs-only PR qualifies, no gate/mission check (`scripts/reconcile-merged-issues.ts:25-29`); and the hosted `roadmap:<id>` rollup counts **any closed** roadmap issue as `done`, so a `wontfix` close increments it (`services/agent-model-proxy/src/github-sync.ts:111-124`). A true M6 signal must additionally verify the PR **passed the required gates** and the closed item is **linked to the vision/roadmap** — that check does not exist and is build-plan item 6. |

**Why M5 ≠ M6 matters (the owner's exact question).** A tick that dispatches nothing exits `0` *identically* to a tick that lands a mission PR (`emit.ts:360-371`). "Running" is cheap to prove (`oa status`, `doctor --live`); "**mission-advancing**" is only measurable at the top of the stack and only for self-driving (roadmap rollup) or OA's own **non-shipped** proof ledger (`check:proof`, `scripts/open-autonomy-proof-audit.ts:3-17` — ships to no profile). So M6 is the maturity model's **missing top rung** for every profile except hosted self-driving.

### Is the model different per profile? Same stages, different signals.

| Signal / requirement | simple-sdlc | simple-gh | simple-gh-sdlc | self-driving (hosted) |
|---|---|---|---|---|
| Substrate | local, local-git, PR-free | local, GitHub | local **or** gh-actions, GitHub | gh-actions, GitHub |
| M3 proof tool | `preflight`+`oa doctor` | `preflight`+`oa doctor`+gh-preflight | local: `oa doctor`; hosted: gh-preflight + a workflow smoke | **hosted target: gh-preflight only — no hosted `doctor`** (`bin/doctor-checks.ts:1040`). *But self-driving declares `targets:[gh-actions, local]` (`ir.yml:17`#pr-141) — a **local-target** self-driving install can use both doctors; only the gh-actions target lacks one.* |
| GH admin / branch protection | none | yes (`["ci"]` is a **placeholder** to replace) | yes (provision.json shipped) | yes (**provision.json now shipped**, pr-141) |
| Model proxy | no | no | gh-actions only | **yes** — funded Cloudflare + OIDC allowlist |
| M4 direction doc | none (standards only) | none | none | **template shipped, unvalidated** |
| M4 work-entry | human ztrack issue + `oa-approved` | plans-as-docs (`ztrack import`) | GitHub Issues + `ready` | strategist→roadmap→planner→issues (seed `items:[]`) |
| M6 mission signal | per-issue `done`+`ztrack check` only | per-issue only | per-issue + merged-PR close (A15) | **roadmap rollup (A16)** |
| Bootstrap difficulty | easiest | low | moderate | hardest (proxy + identity) |

**Reading:** the *skeleton* is profile-independent, but a real IMM validator must carry a **per-profile signal set** — simple-sdlc has no M3-GitHub or M4-vision rung at all; self-driving's M3 can't use `doctor` and its M4 adds "proxy deployed + allowlisted." A one-size checklist would be wrong.

## Q2 — Completion & "running + mission-advancing" criteria, per profile, and what validates them

**"Setup complete AND actually running/mission-advancing"** = the install reaches **M5 (running)** and then **M6 (advancing)**. The concrete, checkable criteria:

**Universal (every profile):**
1. **M2**: `generated.json` valid + all manifest files on disk (A1/A2). *Validated by:* `compile` pre-write gates (deterministic, already fail-closed).
2. **M3 plumbing**: harness committed (A6/A7) + env/deps/auth (A10) + dispatch chain works. *Validated by:* `oa doctor` (local, pr-140) for deps/fence/schedule/prompts/provider — exit 1 on any FAIL, but **WARN never fails**. The **launch chain** is proven only by the *main-tree* `npx open-autonomy doctor --live` (A9, `bin/doctor-checks.ts:1024-1136`) — which costs money, needs a signed-in CLI, is behind the dead npm path, and even then launches a **throwaway probe agent** (not a profile tick). `oa doctor --live` (pr-140) only health-probes the provider.
3. **M4 direction** (profiles that carry one): vision declared + present. *Validated by:* pr-138 preflight hard-FAIL — **but only existence, not content** (a REPLACE-THIS template passes).
4. **M4 board**: ≥1 actionable item under the profile's dispatch predicate (A14). *Validated by:* running the profile's own PM/manager query — no dedicated validator.
5. **M5 running**: fence lifted + a real session launched (A5 + A9/`oa status`).
6. **M6 advancing**: a mission-linked issue closed by a gate-passed merged PR (A15/A16).

**Per-profile completion specifics:**
- **simple-sdlc** — M3 has no GitHub rung; M4 has no vision rung (governance = `standards/*.md`); completion = `oa doctor` green + ≥1 `ready`+`oa-approved` issue + unpaused + a develop→review→done cycle with `ztrack check` green (`profiles/simple-gh-sdlc/skills/pm/SKILL.md:30-32`). No mission rollup — M6 is per-issue only.
- **simple-gh** — add: branch protection with **your real** check names (not the shipped `["ci"]` placeholder, `profiles/simple-gh/provision.json`), a first plans-as-docs cycle producing `ready` issues, and a supervised first manager merge (no auto-merge). 
- **simple-gh-sdlc** — add: `ci`+`agent-review`+`security` required, auto-merge armed *after* a supervised first merge; M6 observable via merged-PR→issue-close.
- **self-driving (hosted)** — add: a deployed, funded, **allowlisted** model proxy; filled constitution/PROJECT/roadmap; `human-approval` in required checks; M6 observable via the `roadmap:<id>` rollup. **No hosted `doctor`** — M3/M5 must be proven by gh-preflight + a first workflow run.

**What validates all this today?** A partial, uncomposed set:
- **`oa doctor`** (pr-140) — deterministic **plumbing** validator (deps/fence/schedule/prompts/skills/provider; `--live` = dispatch). Local-runner only; checks nothing GitHub-side, nothing about the board or the mission.
- **`open-autonomy-preflight`** — GitHub **structural** readiness, but hard-fails only on missing files; the things that actually wedge a hosted install (`MODEL_PROXY_URL`, protection) only **warn** (A12).
- **The `audit` agent** (pr-137) — the closest thing to a **conformance/maturity validator**: nine checks incl. prescription-vs-live branch protection (check 3), gated vision (check 1), fence integrity (check 6), board health (check 9). **But** it is prose-only (no deterministic checker), **dispatch-only** (nothing auto-runs it), **simple-gh-only**, and aimed at **drift in an already-running install** — check 9 assumes a populated board, check 6 assumes a live schedule/provider. It has no "setup complete" mode and no per-profile completion checklist.
- **Mission-advancing:** only self-driving's roadmap rollup (A16) or OA's non-shipped proof ledger (A17). **No shipped ran-vs-advanced signal for simple-*.**

**Does activation fail-closed if setup is incomplete?** Partially — fresh installs land **paused** (A4/A5), the scheduler refuses an uncommitted harness (A6), and a mis-set merge gate **wedges** PRs rather than mis-merging. But **nothing composes the signals into a stage verdict**, nothing forces `doctor`/audit to run, nothing validates the vision *content*, and un-pausing is an unguarded file deletion. **So the honest answer: setup can be "incomplete" (placeholder vision, empty board, wrong check names, undeployed proxy) and still be unpaused into a loop that ticks without advancing anything.** That is exactly the failure mode this design must close.

## Q3 — The one-shot, agent-assisted install (the core design)

**Shape.** A new OA **`install` skill** (the setup agent — the missing product the docs themselves anticipate, `docs/INSTALL-AGENT.md:523`). It is an **agent, not an `oa init` wizard-verb**: per OA doctrine, chaining compile→provision→seed is "scripting what an agent could do," and the agent must adapt to repo shape, draft the vision, and recover from partial failures — so the orchestration lives in the skill, calling the existing verbs (`compile`, `provision-target-repo`, `oa …`) as tools. It runs a **detect → recommend → capture → authorize → execute → validate → hand-off** flow, doing everything mechanical itself and **stopping only at the irreducible human gates**. Target for the local profiles: the user answers **~4 gate questions once**; the agent drives to **M5 (running)** and watches for **M6 (advancing)**.

**Design principle (from OA's own doctrine, `CLAUDE.md`):** *"Never script what an agent can do; a deterministic script is justified by one thing only: security."* So the install agent is an **agent**, not a wizard script — it adapts to repo shape, drafts the vision, recovers from partial failures — while the **security boundaries stay deterministic** (the merge boundary, the pause kill-switch, token minting, the `human-approval` gate). The human gates below are precisely the boundaries an agent must not self-authorize.

### The flow — AGENT-DOES vs ASK-THE-USER

**Phase 0 · DETECT — agent, fully autonomous.** Read the repo without asking: language/build (`package.json`, lockfiles), git remote + whether the repo is empty/new vs populated, `gh auth status` + admin rights + repo visibility/plan, existing `.open-autonomy/` (is this a re-install/upgrade?), tool presence (node≥22.18, tmux, **bun**, ztrack, termfleet, signed-in CLI — mirror `oa doctor`'s env check, A10). *Never ask for what it can read* (`docs/INSTALL-AGENT.md:84-139`).

**Phase 1 · RECOMMEND / CONFIRM PROFILE — agent proposes, user confirms (GATE 1).** Run the **profile recommender decision tree** (`findings-F` Part C) over the Phase-0 facts: local-only→simple-sdlc; existing repo + hosted→simple-gh-sdlc@gh-actions; existing repo + own machine + no-auto-merge→simple-gh; new dedicated repo + can fund a proxy→self-driving. Mechanical inputs it reads: each `ir.yml` `targets`/`codeHost`, presence of `provision.json`, `resources:` containing repo-shell files (⇒ whole-repo scaffold ⇒ new-repo-only, enforced by the clobber guard `bin/autonomy-compile.ts:239-257`). **If the user already picked a profile, skip the ask and just validate the choice against the repo** (warn if e.g. they picked self-driving for a populated repo). Otherwise: **one question** — "I recommend `<profile>` on `<substrate>` because <reasons>; confirm or override?"

**Phase 2 · CAPTURE DIRECTION — agent drafts, user approves (GATE 2).** For any profile that carries a vision/constitution (self-driving today; recommended for all — see build plan): the agent **drafts a proposed North Star + merit criteria** from the repo's README/docs/purpose, then **asks the user to approve or edit** (1–2 questions). This *fills the `<!-- REPLACE THIS -->` templates* (`profiles/self-driving/docs/CONSTITUTION.md:5,12`). **Irreducibly human**: the mission is definitionally the human's; `docs/CONSTITUTION.md` is human-owned ("amended, never auto-edited", `CLAUDE.md`), and #138 auto-gates it into `human_required_paths`. The agent may *propose*, never *decide*, the mission.

**Phase 3 · AUTHORIZE — user consent gates (GATE 3, batched).** Ask, in one batch: (a) **spend** — tick cadence + WIP (local spend is uncapped; `docs/INSTALL-AGENT.md:172-174`); (b) **harness-commit** consent (~40 files incl. the Stop hook); (c) for GitHub — **admin/branch-protection** consent + the repo's **real PR-CI check names** (wrong names deadlock every PR, `docs/INSTALL-AGENT.md:120`) + identity (own token vs bot reviewer); (d) for self-driving — **model-proxy** decision (deploy own vs get allowlisted). These are the human_required consent points; each is a boundary the agent must not self-authorize.

**Phase 4 · EXECUTE — agent, fully autonomous.** In dependency order (`docs/INSTALL-AGENT.md:203` "commit the harness first, wire the gate last"): install deps (bun/termfleet/ztrack, pinned); **`compile`** the profile onto the substrate (A2 gates enforce correctness); **write the filled vision/constitution** from Phase 2; **commit the harness**; bring up the termfleet console+provider on **repo-unique ports** with a pinned `TERMFLEET_PROVIDER_URL`; for GitHub, run **`provision-target-repo`** (branch protection + required checks + seam labels; report secrets as MANUAL); **seed the board with *draft* items** — author the first work item(s) from the captured direction (or dispatch the profile's **planner** once, `oa dispatch planner`, to derive drafts). **It must NOT self-promote items to `ready`/`oa-approved`.** The missing `ready`/`oa-approved` label *is* the day-one dispatch fence, and applying it is a human triage act (`profiles/simple-sdlc/ir.yml:80-83`; pr-139's planners deliberately file drafts, never `ready` — "the missing `ready` label IS the gate", `findings-E`). Promotion belongs to the human at G4.

**Phase 5 · VALIDATE — agent, fully autonomous, fail-closed.** Run the composed **maturity check** (build-plan item 3): `oa doctor` (M3 plumbing — deps/fence/schedule/prompts/skills/provider) + the launch-chain proof (`npx open-autonomy doctor --live`, once 0.4.2 is published — a real launched session, though a *probe* one, not a profile tick) + gh-preflight (M3 structural) + the generalized **`audit` agent** in a new *setup-completion* mode (direction filled + not-placeholder, board seeded with drafts, provision matches live protection). Emit an **IMM stage report** ("reached M4/ARMED; blockers: proxy not allowlisted"). **Do not advance to G4 if any hard signal fails.**

**Phase 6 · HAND-OFF TO RUNNING — user gate (GATE 4) then agent.** Ask the final **unpause / go-live** consent ("a judgment call, not yours to default", `docs/INSTALL-AGENT.md:364`). Then `oa resume` + `oa start` (under tmux/nohup). For GitHub: **supervise the first merge**, then arm native auto-merge (`docs/OPERATIONS.md:513-519`).

**Phase 7 · PROVE ADVANCING — agent, autonomous.** Watch the first wave (`oa status`); confirm a mission-linked issue closed via a gate-passed merged PR (A15/A16) → declare **M6 (ADVANCING)**. If it ticks without advancing (empty board / placeholder vision), loop back and report the specific missing rung.

### Where the human gates sit and why (only four asks)
| Gate | Question | Why it can't be auto-decided |
|---|---|---|
| **G1 Profile** (skip if pre-picked) | confirm/override profile+substrate | Sets substrate, gates, cost model; reversible but foundational. Agent recommends; human owns the call. |
| **G2 Direction** | approve/edit the North Star + merit criteria | The mission is definitionally the human's; constitution is human-owned + `human_required` (#138). |
| **G3 Authorize** (batched) | spend cadence · harness-commit · (gh) admin+real-check-names+identity · (self-driving) proxy | Spend, the merge boundary, and repo admin are security boundaries the agent must not self-grant. |
| **G4 Go-live** | **promote the first item to `ready`/`oa-approved`** · unpause · (gh) approve the first merge, then arm auto-merge | Labeling is the human's triage act (the dispatch fence); the pause is the kill-switch; the first merge + arming auto-merge are the last irreversible authorizations. |

Everything else — detect, recommend, compile, provision, seed *drafts*, validate — is the agent's. **Minimum human touch ≈ 4 gates** — but two honest asterisks the design will not hide: (1) for **self-driving**, "proxy" in G3 is **not one question** — deploying/funding a Cloudflare Worker and getting OIDC-allowlisted is a *sub-project*, so self-driving is not a "~4 questions" install regardless of agent quality; and (2) the **real-CI-check-names** ask on a repo with no open PRs is a *best guess* whose wrong value deadlocks every PR (`docs/INSTALL-AGENT.md:92-94`, `docs/OPERATIONS.md:520-522`) — the agent should open a throwaway probe PR to *read* the real contexts rather than guess. The "few questions" target holds cleanly only for **simple-sdlc / simple-gh(-sdlc)-on-local with a pre-picked or repo-derivable profile**.

## Q4 — Gaps + build plan

### What #135–#141 already provide (don't rebuild)
- **Direction as config** (#138): declare + existence + tamper-gate for vision/constitution/roadmap. *Use as the M4-direction backbone.*
- **`oa` CLI + deterministic `oa doctor`/`oa status`** (#140): the M3/M5 plumbing validator + the operator surface. *Use as the mechanical half of Phase 5.*
- **The `audit` agent's nine checks** (#137): the conformance content. *Generalize + re-point as the maturity validator.*
- **Vision-anchored planners** (#139) + **self-driving `provision.json`** (#141): board self-replenishment + the last missing provisioning manifest. *Use in Phase 4 seeding.*

### What is still MISSING (must build)
1. **A setup/install AGENT** — the orchestrator skill (Phases 0–7). Nothing like it ships; `docs/INSTALL-AGENT.md` is a *playbook*, not an agent, and covers only simple-gh-sdlc-on-local.
2. **A profile RECOMMENDER** — exists only as README prose; needs to be a callable skill/function reading `targets`/`codeHost`/`resources`/`provision.json`.
3. **Vision CONTENT validation** — pr-138 checks existence, not that the `<!-- REPLACE THIS -->` markers were replaced. A one-line check closes the single sharpest correctness gap.
4. **A composed MATURITY VALIDATOR + state record** — an `oa maturity` verb that composes A1–A16 into an IMM stage verdict with a **per-profile signal set**, written to a durable `.open-autonomy/install.json` (there is *no* stage record today; unpause/arm leave no trace).
5. **A "mission-advancing" signal for simple-*** — port A16's `roadmap:<id>` rollup (or a lightweight proof-ledger resource) to the local/simple-gh boards so **M6 is machine-checkable on every profile**, not just hosted self-driving.
6. **An orchestration layer** — compile+provision+seed+schedule are still separate manual commands with no single entry point. Per OA doctrine this should be the **install *agent*** (which adapts/recovers), **not** a deterministic `oa init` wizard-verb; pr-140 pointedly ships no `init`/`up` (its scope is runner extraction, "ergonomics over the file"). So the gap is the agent, not a verb.
7. **Auto-trigger + generalize the `audit` agent** — run it automatically at end-of-install and on a cadence, and extend it beyond simple-gh, with a distinct *setup-completion* mode vs *drift* mode.

### Shortest ordered build plan
1. **(Enabler, do first) Publish 0.4.2 to npm** — until the registry serves 0.4.2, *any* install flow that runs `npx open-autonomy …` fails at compile (npm `latest` = 0.4.1, which crashes; 0.4.2 is a version bump in-repo but was never `npm publish`ed — verified). A release action, not code, but it gates everything downstream.
2. **Vision content gate** *(small, high correctness ROI)* — extend #138's preflight to **WARN** (not hard-FAIL) when a declared `vision`/`constitution` still contains the `<!-- REPLACE THIS -->` markers, and route the real judgment ("is this a *real* North Star?") to the audit agent's check 1/7. A marker tripwire is cheap and trivially gamed, so it *flags* a placeholder install; it does **not** "close" direction validation — OA deliberately made existence, not content, the mechanical bar (`findings-E`), and content is an agent-judgment call.
3. **`oa maturity` + `.open-autonomy/install.json`** — a legitimately *mechanical* readiness probe (precedent: `oa doctor`, `reconcile-merged-issues.ts` "mechanical wiring, not judgment") that composes the deterministic signals (A1–A16 incl. pr-140's `eligibility.ts` for the board rung) into an IMM stage verdict with the per-profile signal set, written to a durable record. Delegates the judgment rungs (direction quality, philosophy conformance) to the audit agent. Gives every later phase one "what stage am I at" call and the first durable stage record.
4. **Generalize + auto-wire the `audit` agent** — all profiles; add a *setup-completion* mode (direction filled, board seeded, provision==live protection, first-tick smoke) distinct from its *drift* mode; auto-dispatch at end-of-install and on cron.
5. **Profile recommender skill** — the decision tree (`findings-F` Part C) as a callable reading the mechanical inputs (`targets`/`codeHost`/`resources`/`provision.json`); feeds the install agent (and usable standalone).
6. **The `install` setup agent** — the orchestrator skill running Phases 0–7, pausing only at G1–G4, calling the existing verbs as tools (no new `oa init` wizard-verb). Ship as an OA skill (per `docs/INSTALL-AGENT.md:523`).
7. **A real mission-advancing signal** — a check that a closed work item was closed by a **gate-passed** merged PR **linked** to the vision/roadmap (today's `reconcile-merged-issues`/roadmap rollup count docs PRs and wontfix closes, `findings-F` A15/A16), ported to non-hosted boards so M6/ADVANCING is provable on every profile.

**Bottom line.** After #137–#141, OA has the *ingredients* of an agent-assisted install — a validated direction layer, a deterministic runner CLI, a conformance-audit doctrine, self-replenishing planners, and (finally) a provision manifest for every GitHub profile — but **no cook**: no install agent, no profile recommender, no composed maturity verdict, no vision-content check, and no ran-vs-advanced signal off the hosted path. The build plan above turns those ingredients into the one-shot install the owner wants, and the IMM (M0→M6) gives it the stage model — profile-parameterized, not one-size — to know when the job is actually done: **not at "it ran," but at M6, "a gated merge advanced the mission."**
