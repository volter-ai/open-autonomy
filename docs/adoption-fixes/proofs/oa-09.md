## 3. Per-assertion mapping to AC-7 + reproduction + pointer to the Tier-A proof

AC-7 (spec, verbatim): *"On a box with only a foreign provider live on 7373: a fresh install following the
new quickstart (unique ports + pin) runs `node scheduler/run.mjs --once` and every launched session lands
on the pinned provider ... the foreign provider on 7373 shows none."*

`packages/substrate-local/src/provider-landing.test.ts` maps onto this as follows:

| Test | Claim proven | Key assertions |
|---|---|---|
| `AC-3: ... a hello install compiled with --provider-url carries TERMFLEET_PROVIDER_URL` | The durable pin lands in `scheduler/schedule.json.env` for a REAL compiled+materialized install (not just an in-memory `compileLocal` call). | `schedule.env` equals `{ TERMFLEET_PROVIDER_URL: 'http://127.0.0.1:7602' }`. |
| `AC-7: pinned + a foreign current-context decoy on 7373 present: the launched session lands on 7602, 7373 is never touched, log says (schedule)` | **The heart of AC-7.** A REAL tick (`node scheduler/run.mjs --once`) runs the REAL dispatch chain end-to-end with a live foreign-provider decoy staged, and: (a) the pin lands — every `resolveDefaultProvider()` call resolved to 7602; (b) the actual launched (stub) session's own record shows `provider: "http://127.0.0.1:7602"`; (c) the foreign decoy is never touched — no sink line, no session, and the string `"7373"` never appears in any diagnostic output; (d) the effective-source log agrees at BOTH levels of the dispatch chain — `[loop] provider ... (schedule)` (the loop driver, AC-4) AND `[runner] provider ... (schedule)` (the nested backend resolve, proving the pin held all the way down, not just at the loop's own guess). | `sinkLines.every(l => l === PINNED_URL)`; `sinkLines.some(l => l.includes('7373')) === false`; `sessionLines.length === 1`, `sessionLines[0].provider === PINNED_URL`; `sessionLines.some(s => s.provider === FOREIGN_URL) === false`; both log-line regexes match. |
| `CONTRAST: the SAME foreign current-context decoy, UNPINNED — the launched session lands on the foreign 7373` | Proves the decoy is a REAL, non-vacuous candidate, not dead code — the identical fixture, unpinned, DOES land on 7373. Without this test the AC-7 test above would be unfalsifiable (nothing would prove the extended stub's current-context fallback is actually wired up). | `sinkLines.every(l => l === FOREIGN_URL)`; `sessionLines[0].provider === FOREIGN_URL`; log shows `(current-context)`. |
| `a genuine ambient TERMFLEET_PROVIDER_URL overrides the compiled pin — lands on the AMBIENT url, never 7602 nor the 7373 decoy` | Precedence strengthening, WITH the real-landing assertion `provider-pin.test.ts` cannot make (it only ever observes the child's inherited env via a stub `run-agent.mjs` printing `CHILD_PIN=...` — it never calls the termfleet SDK). | Session lands on `9999`; sink never contains `7602` or `7373`; log shows `(env)`. |
| `BLOCKER-2 regression, on the REAL landing: a set-but-EMPTY ambient TERMFLEET_PROVIDER_URL does NOT drop the pin` | Same strengthening for the empty-ambient-idiom regression `provider-pin.test.ts` already covers at the child-env level. | Session still lands on `7602`; log still says `(schedule)`. |

## 4. HONEST BOUNDARY

This drives OA's own routing/launch layer end-to-end and model-free via the stub provider
(`packages/substrate-local/src/test-support/stub-termfleet.ts`, extended — see below) — it lands a real
window on the RESOLVED (pinned) provider and proves the foreign 7373 decoy gets none. **It does NOT
exercise termfleet's real virtual-tmux provider process** — a real termfleet provider paired with a real
coding-CLI would spawn a billed agent, which is forbidden (see the owner's live-agent-test-safety rule:
never pair a real termfleet provider with a real coding-CLI in a test). Zero model calls anywhere in this
file or its fixtures. Do not read this as "a real termfleet process was walked end-to-end" — it was not;
the SDK surface (`ProviderClient`, `providerRefFromUrl`, `resolveDefaultProvider`) is a functionally real
but model-free/network-free stand-in, exactly the same posture OA-08's `launch-verification.test.ts` and
OA-18's `bin/doctor.test.ts --live` suite already use for the identical reason.

**Why `stub-termfleet.ts` was extended, and why the extension lives in the NEW test file, not the shared
one:** the shared stub's `resolveDefaultProvider` only ever looks at `opts.url` — it has no notion of
`~/.termfleet/current.json` (the real SDK's current-context fallback). Without that fallback, staging a
"foreign provider on 7373" fixture would be **vacuous**: nothing in the resolution path would ever consult
it, pinned or not, so "the foreign provider gets none" would trivially always pass even with the pin
removed. `provider-landing.test.ts` writes its OWN fixture-scoped overrides of
`node_modules/termfleet/index.js` (adds a `provider` field to each recorded session — the "landed on"
evidence) and `node_modules/@termfleet/core/local-providers.js` (restores the real precedence: explicit
url > `~/.termfleet/current.json` > auto-local) directly into each test's own temp fixture, on top of
`installStubTermfleet()`'s baseline files. **The shared `stub-termfleet.ts` file itself was never edited** —
confirmed by `git diff --stat adoption-fixes-backlog..build/oa-09` above (one file changed, the new test).
Every existing importer (`packages/substrate-local/src/launch-verification.test.ts`, `bin/doctor-checks.test.ts`,
`bin/doctor.test.ts`) is unaffected; `provider-pin.test.ts:262-313` is cited, not modified.
