# OA-18 proof — `open-autonomy doctor`'s AC-6 / AC-10 "live-pending" gap, closed

Spec: `docs/adoption-fixes/OA-18-doctor-self-verifying-install.md`. OA-18 landed code-complete, but two
acceptance criteria — AC-6's `--live` captured-login-prompt path, and AC-10's live-tick pass/fail paths —
were left **live-pending**: the box they were built on had no termfleet+model stack to drive `checkLive`
(`bin/doctor-checks.ts:1024-1136`) against. This proof closes both as **committed, deterministic tests**
that drive the REAL `checkLive` code against a **model-free stub termfleet provider** — zero model calls,
zero spend, no real coding-CLI, and (the whole point) durable: it survives a box wipe because it lives in
git, not in a throwaway environment.

This file replaces an earlier, since-wiped `/workspace/proofs/oa-18.md` — the durability requirement this
proof exists to satisfy.

## 3. What each AC now proves, and how to reproduce

**AC-6** (F-13 — auth advice must never be `--version`, and `--live` proves sign-in for real):
`bin/doctor.test.ts`'s `'AC-6: on a box where the coding CLI is signed out, doctor --live exits non-zero
naming the captured login prompt (independently of the static auth FAIL)'` scaffolds a healthy compiled
install, installs the stub termfleet, PATH-shims a `claude` binary whose `auth status` reports "not logged
in" (exit 1), sets `OA_STUB_TF_DIE=1` + `OA_STUB_TF_CAPTURE=<a login-prompt string>`, then runs
`node dist/cli.js doctor --live --json`. It asserts BOTH layers: the static `auth` check FAILs
("NOT signed in", never citing `--version`) AND the `live` check FAILs, embedding the exact captured prompt
in its detail — and the process exits non-zero. `bin/doctor-checks.test.ts`'s companion unit test
(`'AC-6 (--live): a logged-out coding CLI dies at launch on a captured login prompt...'`) proves the same
thing one layer down, calling `checkLive` directly. The pre-existing signed-out **unit** path
(`bin/doctor-checks.test.ts`'s `checkAuth` describe, unchanged) stays green throughout — confirmed by the
full `check:doctor` run above.

Reproduce:
```
bun test bin/doctor.test.ts -t "AC-6"
bun test bin/doctor-checks.test.ts -t "AC-6"
```

**AC-10 pass path** ("On a fully healthy install, `doctor --live` launches exactly one session through
`run-agent.mjs`→`autonomy-runner.mjs`, reports it survived (or emitted `DOCTOR-OK`), cancels it, exits 0"):
`bin/doctor.test.ts`'s `'AC-10 pass: a healthy install -> doctor --live launches exactly one (stub) session
that survives, exits 0'` scaffolds+commits a real `simple-sdlc local` install, installs the stub termfleet,
PATH-shims a signed-in `claude`, and runs `doctor --live --json` with `OA_STUB_TF_SESSIONS_FILE` pointed at
a real file. It asserts `live.status === 'PASS'`, the detail names `run-agent.mjs`, `verdict === 'PASS'`,
`r.status === 0`, AND that the sessions file was actually written (a real `createAgentWindow` call
happened — not a vacuous pass). `bin/doctor-checks.test.ts` has two unit companions: one proving the
genuine survival-through-polling mechanism (capture unset, so a DOCTOR-OK shortcut can't be the reason it
passed), and one proving the `DOCTOR-OK`-emission alternate path the spec also accepts.

Reproduce:
```
bun test bin/doctor.test.ts -t "AC-10 pass"
bun test bin/doctor-checks.test.ts -t "AC-10 pass"
```

**AC-10 fail path** ("on a repo in state 7 or 8, `doctor --live` exits non-zero and its output includes
the captured terminal contents of the dead session"): `bin/doctor.test.ts`'s `'AC-10 fail: a session that


## 4. Honest boundary

This drives the REAL OA dispatch/capture layer — `scripts/run-agent.mjs`, `scripts/autonomy-runner.mjs`,
`backend.mjs`'s `TermfleetRunner`, and `bin/doctor-checks.ts`'s `checkLive` itself — **model-free**, via a
stub termfleet provider (`packages/substrate-local/src/test-support/stub-termfleet.ts`). It does **not**
exercise termfleet's real virtual-tmux process, its real socket.io transport, or a real coding-CLI session.
Concretely, the stub's `ProviderClient.createAgentWindow`/`snapshot`/`captureTerminal` are hand-written JS
returning canned, env-var-driven values — not termfleet's actual tmux-window lifecycle. A regression in
REAL termfleet's own behavior (a socket.io protocol change, a real tmux capture bug, a real provider
crash) would not be caught by this suite; that gap is unchanged from before this proof and is explicitly
out of scope for OA-18 (which proves OA's own dispatch/capture code, not termfleet's implementation).

Per the owner's "0 reasons to run claude" directive: every test in this proof makes **zero model calls**.
No real `claude`/`codex` binary is ever invoked in a mode that could spend (the `claude` PATH shims used
for the `auth` check are pure shell scripts printing canned text — `echo ... exit 0/1` — never invoking a
real CLI), and no real termfleet provider process is ever started. This is verifiable by grepping either
test file: no `ANTHROPIC_API_KEY`, no real `claude`/`codex` binary path, no network call.
