# Organization compiler API 1.0

The public library surface is exported from `@open-autonomy/core` and pinned by `COMPILER_API_VERSION`. `CompilerArtifact` is the immutable, content-addressed wire boundary. Its digest covers the protocol version, operation, semantic content kind, level, producer, input digests, media type, and content; annotations such as wall time are deliberately absent. Kinds distinguish organizations, resolved/normalized organizations, analysis reports, deployment searches, plans, observations, event batches, and histories, so equal compiler levels never imply substitutable content.

`runStableCompiler` executes a bounded linear pass chain. `runCompilerBuild` executes an artifact DAG: analyses branch into non-substitutable reports, solve retains the normalized organization beside its deployment search, and lowering carries the selected candidate and organization through control and execution bundles. Lift/replay graphs may start from native observations rather than pretending every operation is one forward pipeline. Ready nodes and merged results are sorted by node id, independent of completion timing. Clean, incremental, warm-cache, serial, and parallel executions must yield identical artifact bytes.

The stable operation vocabulary is `parse`, `link`, `normalize`, `analyze`, `solve`, `lower`, `emit`, `lift`, and `replay`. Their closed audited adapters invoke the existing typed domain functions: artifact/Organization parsing, module resolution, organization normalization, bounded organization analysis, deployment solving, control/execution/native lowering and emission, declarative native-observation lifting, and causal-history replay. DAG execution accepts only these identifiers plus deterministic artifact projection/diagnostic built-ins; callers cannot inject JavaScript callbacks into the compiler realm. Stage contracts enforce legal operation and level transitions, and the artifact DAG keeps side analyses and reverse observational flows distinct.

In-process passes must declare zero ambient capabilities. IO-bearing third-party plugins use `runIsolatedPlugin` and a host executor. `BubblewrapPluginExecutor` clears the environment, unshares namespaces, binds only declared read/write paths, denies secrets, bounds wall time/output, and enables networking only when explicitly declared. A host that cannot provide the requested isolation must fail closed; running arbitrary plugin code in the compiler realm is nonconforming.

Diagnostics are control-escaped, redacted, and bounded before streaming, while fatal control state is tracked outside the bounded transcript. Abort signals, wall/input/output/diagnostic/pass/parallelism bounds are explicit request inputs. Cache keys cover API version, pass identity/version/configuration/capabilities, and input digest. Cache entries require an HMAC from a compiler-held authentication key and are then digest- and provenance-verified before use; storage cannot substitute a merely self-consistent artifact.

The CLI `bun bin/organization-compiler-api.ts` reads one bounded JSON request from stdin. `wrap` creates the exact library artifact envelope and `verify` validates an envelope. This deliberately small transport surface makes golden CLI/library equivalence testable; graph construction and capability brokers remain typed library concerns rather than executable code embedded in JSON.

## Compatibility policy

API and artifact protocol versions use semantic versioning independently of the CLI package. Patch releases preserve accepted request/artifact meaning. Minor releases may add optional fields and operations but accept the previous two minor versions. A major version may reject old artifacts only after a published migration and at least two minor releases of deprecation. Unknown major versions fail closed. Cache entries always pin the exact API, pass version, and artifact digest, so upgrading cannot silently reinterpret stored output.
