## 3. Per-AC assertions, and how to reproduce

Fixture (`scaffold()` in `pm-dispatch-fence.test.ts`): a real `compileLocal(ir)` output (pm
+ develop agents, `codeHost: 'github'`, `policy.box.dispatch: { mode: 'allowlist',
allow_label: 'oa-approved' }` — mirrors `profiles/simple-sdlc/ir.yml`'s own landed policy
box), written to a fresh `mkdtemp` dir, the shared stub termfleet installed, a REAL `ztrack
init --preset simple-sdlc`, then `git init` + `git add -A` + `git commit` — a real git repo
with a real committed ztrack board. Unlike OA-08's AC-7 fixture, the `develop` skill is
**left intact** here (a stub `SKILL.md`, same as every other agent's copy) — this suite is
about *which* issues get dispatched, not whether a dispatch can succeed at all, so a
successful dispatch must be observable.

Each AC calls `runDispatchFenceTick()` (`test-support/canned-pm-dispatch-fence.ts`), which:
- reads `policy.dispatch` from a REAL `.open-autonomy/autonomy.yml` (parsed off disk, never
  cached);
- reads the board via a REAL `ztrack issue list --state open --limit 100 --json
  identifier,title,state,labels,assignee` (pm/SKILL.md:58's own command);
- evaluates the allowlist gate over the **whole** `ready` set first (pm/SKILL.md:18-23 is a
  blanket statement — "regardless of its `ready` state" — true independent of board-list
  position; see the driver's own jsdoc for why this is a deliberate two-pass design, not an
  arbitrary implementation choice: ztrack's list order was empirically observed to reorder
  on label/body edits, e.g. most-recently-touched first, so relying on scan order for the
  allowlist check would make the test's pass/fail depend on incidental ordering);
- for every allowlist-cleared candidate, in board-list order, reads its body via a REAL
  `ztrack issue view <id> --json` and scans for a deferral marker (pm/SKILL.md:85-93);
- dispatches the REAL `bun scripts/runner.ts launch develop --ref <id> --branch
  agent/issue-<id>` for the first candidate that clears both gates, and stops (one
  dispatch per tick, per pm/SKILL.md:134).

**AC-7** (`pm-dispatch-fence.test.ts`, "OA-07 AC-7" describe block):
- Board (ordering is load-bearing — see the non-tautology note below): the approved issue
  is created AND labeled `oa-approved` (a REAL `ztrack issue edit --add-label oa-approved`)
  FIRST, then the unlabeled `ready` issue is created LAST. Because ztrack lists
  most-recently-touched first, the **unlabeled issue is the board's TOP candidate**. The
  test asserts this explicitly (`readBoard(dir)[0] === <unlabeled>`) so the fixture's premise
  can't silently regress.
- `tick.fenced` === `[{ id: <unlabeled>, note: 'fenced (no oa-approved)' }]` — the exact
  token shape pm/SKILL.md:21 specifies.
- `tick.blockedForHuman` === `[]` — a fenced issue is explicitly NOT reported
  blocked-for-human (pm/SKILL.md:21-22: "it's not a decision, it's simply not yet opted
  in").
- `tick.dispatched` === `[<approved>]`; `tick.launch.status === 0` — the REAL runner
  accepted this dispatch. This is a genuine dispatch-outcome assertion: the top candidate
  (unlabeled) was fenced out of pass 2, so the approved issue is the one that actually got
  launched.
- Independent proof (not the driver's own return value): the shared stub provider's
  `OA_STUB_TF_SESSIONS_FILE` contains **exactly one** session record, `agent: 'develop'`,
  whose `cwd` contains `agent-issue-<approved>` and does **not** contain
  `agent-issue-<unlabeled>` — attribution proven by inspecting the sessions file, not
  merely trusting the tick output.
- The fenced issue never got a worktree or branch at all (`git rev-parse --verify
  agent/issue-<unlabeled>` fails; the `.worktrees/agent-issue-<unlabeled>` directory does
  not exist) — it was never launched, full stop.
- **Non-tautological (and specifically defended against the "wrong item sorts first"
  trap)**: because the unlabeled issue is the board's TOP candidate, with the allowlist gate
  removed it WOULD be body-read and dispatched FIRST — it would land the one session and the
  approved issue would never be reached. This reproduces the real F-7 audit scenario (a
  pre-existing, un-opted-in item dispatched in preference to the approved one). The gate is
  therefore load-bearing on the actual **dispatch outcome** (the `tick.dispatched` /
  sessions-file / worktree assertions), not merely on the self-reported `fenced` note — as
  the §4 probe below demonstrates by literal removal.

**AC-8** (`pm-dispatch-fence.test.ts`, "OA-07 AC-8" describe block):
- Board: one `ready`, `oa-approved` issue whose body reads (in full) "Do not dispatch —
  deferred per decision record OA-DR-12." plus surrounding context — the spec's own example
  marker shape.
- **Tick 1**: `tick1.fenced === []` (it has the label — never a fence hit);
  `tick1.blockedForHuman === [{ id: <deferred>, note: 'blocked-for-human (body: explicit
  deferral marker)' }]`; `tick1.dispatched === []`; `tick1.launch === null` — **no
  subprocess is spawned at all**, not merely one that fails. The stub provider's sessions
  file has zero records; no worktree/branch was created for the issue.
- **Tick 2** (a fresh call to `runDispatchFenceTick`, re-reading the board and the body from
  scratch — no in-memory state carried over, matching "there is no separate 'already saw
  this' memory to maintain"): identical result — `blockedForHuman` still names the same
  issue, `dispatched` still `[]`, `launch` still `null`, sessions file still empty. This is
  the AC's own "repeat tick → still not dispatched" requirement, proven by literally calling
  the tick function twice and asserting the same outcome both times.
- **Non-tautological**: the issue is `ready`, `oa-approved` (clears the allowlist cleanly),
  and has no branch — with the body-read removed it WOULD be dispatched (again, verified
  live — see §4).

**Reproduce:**

## 5. HONEST BOUNDARY

This proves the **mechanism half** of both ACs — the REAL `policy.dispatch` read, the REAL
board read, the REAL per-issue `ztrack issue view` body read, and the REAL `bun
scripts/runner.ts launch develop` dispatch landing a session for exactly the eligible
issue (proven by the stub provider's own session file, not just the driver's return value)
— deterministically and durably (it survives a full repo wipe: nothing here depends on a
live process, termfleet, or any external service; the only non-builtin dependency is the
repo's own pinned `ztrack@1.0.0` devDependency).

It does **NOT** prove that a real-model PM, handed only `SKILL.md` and the board each fresh
tick, independently **reads** the doctrine and **decides** to take these exact actions —
in particular, a model's own reading comprehension of an arbitrary prose deferral marker
("Do not dispatch", "blocked by decision X", a paraphrase neither test author anticipated,
…) is inherently a live, model-judgment question that no fixed regex or canned driver can
stand in for. That judgment is canned in
`test-support/canned-pm-dispatch-fence.ts`'s `runDispatchFenceTick` — its function body is
a direct, literal transcription of pm/SKILL.md's allowlist-gate and body-read rules
(:18-23,79-80,85-93), not independently derived by a model. This is consistent with:
- the spec's own AC-7/AC-8 text ("Live tick test" — judgment-layer, verified behaviorally,
  never claimed as unit-testable);
- OA-08's own AC-7 honest boundary (`docs/adoption-fixes/proofs/oa-08.md` §4), which draws
  the identical mechanism-vs-judgment line for the sibling escalation doctrine;
- `CLAUDE.md`'s "Built vs designed" section, which names the sibling case (escalate-on-SLA)
  as a live-proven `test.todo`, never a unit test.

**Named residual gap:** a live/bench run of the ACTUAL `pm` skill (model-driven), on a
populated board carrying both a fenced and an approved issue (AC-7) and separately on a
board carrying a body-deferred issue re-ticked twice (AC-8), reproducing these exact
outcomes — fence the unlabeled one, dispatch only the approved one; refuse the deferred one
on tick 1 AND tick 2 — remains the only proof of the judgment layer itself. This file is
deliberately **not** that proof and does not claim to be. It closes the mechanical half of
both ACs (the half that doesn't need a model to be true) as a committed, repeatable,
durable test; the judgment half stays exactly where the spec always said it belonged — live
verification.
