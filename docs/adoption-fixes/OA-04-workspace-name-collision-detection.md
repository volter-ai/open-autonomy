# OA-04: Detect and fail loudly on npm-workspace / package-name collisions with the runner's dependency tree

**Finding:** F-4 — npm-workspaces / package-name collisions are fatal and unhandled: a host workspace package silently shadows a published termfleet/OA dep, and the emitted runner's bare `import 'termfleet'` self-resolves to the host repo itself when the host package is named `termfleet` (see OA-INSTALL-AUDIT-FINDINGS.md §2)
**Priority:** P0
**Fix target:** open-autonomy

## Problem

Adopting open-autonomy into an existing JS repo puts the emitted runner's dependencies into the **host repo's dependency namespace**. Two distinct failure modes were observed on the audit target (a clone of `volter-ai/termfleet` — an npm-workspaces monorepo whose own `packages/core` is `@termfleet/core` and whose root package is named `termfleet`):

- **(a) Workspace-link shadowing.** npm workspaces symlink `node_modules/@termfleet/core` → `packages/core` (the repo's in-development source), silently shadowing the published `@termfleet/core@0.2.1` that the published `termfleet@0.2.0` CLI/SDK requires. The published CLI crashed with `ERR_PACKAGE_PATH_NOT_EXPORTED: './agent-launch.js' is not defined by exports in @termfleet/core/package.json`. There is **no supported npm mechanism** to prefer the registry package over a workspace link (`--install-links` applies to `file:` deps, not workspaces).
- **(b) Node self-reference resolution.** The emitted runner (`scripts/autonomy-runner.mjs` in an install) imports the bare specifier `termfleet`. When the host repo's root `package.json` is named `termfleet` and carries an `exports` field, Node's ESM self-reference resolution binds the bare specifier to the **host repo itself** — *even when a genuine published copy is installed at `node_modules/termfleet`* (empirically verified, Node v22.23.1). In an unbuilt repo this crashes with `Cannot find module <repo>/dist/index.js`; in a **built** repo it would **silently run the host's dev code as the runner SDK** — the worst outcome, because nothing crashes.

Today nothing in the toolchain detects either mode: `preflight` (bin/preflight.ts) checks only node-pty and the lockfile; `compile` (bin/autonomy-compile.ts) checks only file clobbers; the emitted loop driver's guard checks only that `node_modules/termfleet` *exists* — which it does in the self-reference case, so the guard passes and the crash happens several process-hops deep. Docs never mention the word "workspace". The extreme form here (the target *is* termfleet) is target-specific, but the class is not: any monorepo hosting a package whose name collides with anything in the runner's dependency closure (e.g. a fleet repo with a workspace named `ws`, `zod`, `react`, `@termfleet/*`, `ztrack`, …) hits the same wall.

## Root cause (verified file:line citations)

- **The emitted runner imports its SDK by bare specifier from the host's namespace.** `packages/substrate-local/src/backend.mjs:15-16`:
  ```js
  import { ProviderClient, providerRefFromUrl } from 'termfleet';
  import { resolveDefaultProvider } from '@termfleet/core/local-providers.js';
  ```
  This file is emitted **verbatim** into every install as `scripts/autonomy-runner.mjs`: `packages/substrate-local/src/emit.ts:34` (`RUNNER_BACKEND = readFileSync(join(here, 'backend.mjs'), 'utf8')`) and `emit.ts:239` (`generated['scripts/autonomy-runner.mjs'] = RUNNER_BACKEND`). The header comment at `backend.mjs:8` states the contract — "The install must have `termfleet` (+ `@termfleet/core`) in node_modules" — but nothing verifies that those names *resolve to the published packages* rather than to host source. The dev-time TS twin has the same imports (`packages/substrate-local/src/runner.ts:8-9`).
- **The only existing guard tests existence, not resolution.** The emitted loop driver's termfleet guard, `packages/substrate-local/src/emit.ts:83-90` (inside the `LOOP_DRIVER` template emitted as `scheduler/run.mjs`):
  ```js
  if (needsRunner && !existsSync(join(here, '..', 'node_modules', 'termfleet'))) { … exit(1) }
  ```
  Verified insufficient: in a reproduction repo (root `package.json` named `termfleet` with `exports`, workspace `packages/core` = `@termfleet/core@0.2.0`, published `termfleet@0.2.0` npm-installed), `node_modules/termfleet` **exists as a real registry copy**, so the guard passes — yet `import.meta.resolve('termfleet')` from `scripts/` resolves to `<repo>/dist/index.js` (self-reference) and the import fails `ERR_MODULE_NOT_FOUND`, exactly the audit's tick-1 crash. Its test (`packages/substrate-local/src/scheduler-termfleet-guard.test.ts`) covers only the not-installed case.
- **No preflight or compile-time check exists.** `bin/preflight.ts:111-115` runs exactly two checks (`ensureNodePty()`, `verifyLock()`) — no namespace/collision check. `bin/autonomy-compile.ts:92-107` guards only byte-level file clobbers (`findClobbers`) before `materialize`; it reads nothing from the target repo's `package.json`.
- **Verified collision mechanics** (scratchpad reproduction, Node v22.23.1 / npm 11.12.1):
  - `node_modules/@termfleet/core` → symlink `../../packages/core` (`realpath` escapes `node_modules` into the repo tree) — the workspace shadows the published dep termfleet needs.
  - `import.meta.resolve('termfleet')` from a file under `scripts/` → `file://<repo>/dist/index.js` despite `node_modules/termfleet` being a real published copy — self-reference wins over `node_modules` when the enclosing package scope's name matches and it has `exports`.
  - `createRequire(<repo>/node_modules/_anchor.js).resolve('termfleet')` → `<repo>/node_modules/termfleet/dist/index.js` (anchoring inside `node_modules` defeats self-reference) **but** `resolve('@termfleet/core')` still lands in (or fails on) the workspace link — anchoring does **not** fix shadowing.

## Proposed fix

Three layers; the first is the fix the AC gates on, the third is hardening.

### 1. Preflight collision detection (primary — new check in `bin/preflight.ts`)

Add a `checkNamespaceCollisions()` check, run **before** the existing checks in the driver at `bin/preflight.ts:111-115` (collisions explain downstream failures, so they must print first). Implement in a shared module (e.g. `bin/collision-check.ts`) so compile can reuse it (§2). Logic, all under the adopter repo's cwd:

1. **Build the protected name set:**
   - *Direct set (static):* the bare specifiers the emitted runtime and install flow bind at runtime: `termfleet`, `@termfleet/core` (backend.mjs:15-16), `ztrack` (project dep; the installed preset `import`s `ztrack/preset-kit` — docs/OPERATIONS.md:199-201), and `open-autonomy` (a local workspace of that name would shadow every `npx open-autonomy` invocation via `node_modules/.bin`).
   - *Transitive closure (dynamic):* BFS over `dependencies` starting from `node_modules/termfleet/package.json` and `node_modules/ztrack/package.json` when present, reading each dep's installed `package.json` (nested `node_modules` fallback). This is read at runtime — never a hardcoded list — so it tracks whatever termfleet actually ships (24 deps today: `@homebridge/node-pty-prebuilt-multiarch`, `ws`, `zod`, `react`, …). If termfleet isn't installed yet, fall back to the direct set and note it.
2. **Check A — self-reference hazard:** read the host root `package.json` `name`. If it is in the *direct set* → **error** (hard fail), explaining Node self-reference (name match + `exports` binds the bare specifier to the repo itself; flag even without `exports` — adding one later breaks silently).
3. **Check B — workspace shadowing:** expand the host `package.json` `workspaces` globs; collect each member's `name`. A member name in the **direct set** → **error**. A member name only in the **transitive closure** → **error** too (same failure class: the runner would load host dev source as a production dep), with the owning chain printed (e.g. `ws ← termfleet`).
4. **Check C — resolution probe (authoritative, catches anything A/B missed):** for each direct-set specifier that should be installed, resolve it the way the emitted runner will — `spawnSync('node', ['--input-type=module', '-e', "console.log(import.meta.resolve(process.argv[1]))", name], { cwd })` (cwd shares the root package scope with `scripts/`, so self-reference behavior is faithful). **Error** if: (i) resolution fails; (ii) the resolved path is not under `<cwd>/node_modules/`; or (iii) `realpathSync` of the package dir escapes `node_modules` into the repo tree (a workspace/file link).
5. **Error text (named + actionable):** name the colliding package, the mechanism (self-reference vs workspace link), what would happen (silently running the repo's dev code as the runner SDK, or a deep `ERR_MODULE_NOT_FOUND`), and the remediations — rename the colliding workspace/root package, or run the OA loop from a repo that does not develop the runner's own dependencies — and state explicitly that **npm has no flag to prefer the registry copy over a workspace link**, so there is no in-place override.

### 2. Compile-time check (`bin/autonomy-compile.ts`)

When an `outDir` is given (the materialize path entered at `bin/autonomy-compile.ts:92`) and `<outDir>/package.json` exists, run the same shared check against `outDir` **before** `materialize` (i.e. beside the clobber guard at :96-106). Hard-fail with the same named error; allow `--force` to proceed (mirroring the existing clobber-guard escape hatch, :102). This satisfies the "compile in a colliding repo fails loudly" half of the AC even for operators who skip preflight. Note: at compile time termfleet may not be installed yet, so this layer typically exercises checks A/B (static names) only — that is exactly the case that matters (host named `termfleet`, workspace named `@termfleet/core`).

### 3. Harden the emitted runner's own resolution (evaluated; pick + justification)

**Primary recommendation: upgrade the emitted loop driver's guard from an existence check to the resolution probe** (same probe as Check C), replacing `packages/substrate-local/src/emit.ts:84`'s `existsSync(join(here,'..','node_modules','termfleet'))`. On probe failure, print the same named collision error and `exit(1)` before any tick fires. Rationale: preflight is skippable and dependency trees drift after install (a workspace added later re-introduces the collision); the loop driver is the last chokepoint before the failure goes several process-hops deep (`run.mjs → run-agent.mjs → autonomy-runner.mjs → import 'termfleet'`, per `scheduler-termfleet-guard.test.ts:1-4`). It is ~10 lines in the `LOOP_DRIVER` template, changes no install contract, and converts both silent-shadowing and self-reference into the same loud, named, pre-spend error. Detection-and-refuse (not auto-repair) is the honest behavior because npm offers no override; extend `scheduler-termfleet-guard.test.ts` with a colliding-workspace fixture.

**Also do (docs):** add a "npm workspaces / name collisions" caveat paragraph to `docs/OPERATIONS.md` § Local-runner quickstart (after the overlay note at :41-46) and to `docs/INSTALL-AGENT.md` Phase 1 DETECT (the JS-repo detection block at :80-81 should also detect `workspaces` in package.json and cross-check names) — today neither doc contains the word "workspace".

## Alternatives rejected

- **Anchored `createRequire` in `backend.mjs` as the fix** (resolve `termfleet` from a path inside `node_modules`). Empirically it defeats self-reference for `termfleet` but does **nothing** for workspace-link shadowing of `@termfleet/core` (or any transitive dep) — `node_modules/@termfleet/core` *is* the workspace symlink, and every resolution from anywhere inside the repo finds it. Worse, it would fix only the half of the failure class that crashes loudly today while leaving the half that silently runs dev code — masking the symptom that leads an operator to the real problem. Kept out; the resolution probe (which detects rather than papers over) subsumes its value.
- **A dedicated package boundary for the runner's deps** (emit `scripts/package.json` with `{"name":"oa-local-runner", "dependencies":{"termfleet":…}}` and install into `scripts/node_modules` via `npm install --prefix scripts`). This structurally fixes *both* modes (the enclosing scope is no longer the host package, and `scripts/node_modules/termfleet` wins the walk-up before the host's link is reached) and is the right architectural end-state. Rejected *for this fix* because it changes the documented install contract (where deps live, a second lockfile, the `PATH`/`npx` assumptions in `runner-frontend.ts:45` and docs), which is a larger, riskier change than a P0 unblocking fix should carry. Record it as the follow-up architecture item; the shared collision-check module and probe remain useful under it.
- **Docs-only warning.** F-4 is a hard-stop with "no supported workaround inside the repo"; a paragraph nobody reads at install time doesn't meet the AC ("named, actionable error"). Docs are included, but only as reinforcement.
- **Auto-renaming / auto-fixing the host's packages.** Mutating the host repo's identity is far beyond the installer's remit (cf. F-17) and could break the host's own build.

## Acceptance criteria (numbered, testable, fail-today/pass-after, exact command/test)

Fixture (used by 1-4): `REPO=$(mktemp -d)`; in `$REPO` create `package.json` = `{"name":"termfleet","version":"0.0.0-dev","exports":{".":"./dist/index.js"},"workspaces":["packages/*"],"dependencies":{"@termfleet/core":"^0.2.0"}}` and `packages/core/package.json` = `{"name":"@termfleet/core","version":"0.2.0","exports":{".":"./dist/index.js"}}`; run `npm install && npm install termfleet` (both succeed; `node_modules/@termfleet/core` is a workspace symlink — the audit repo's exact shape).

1. **Preflight names the collision and fails.** `cd $REPO && npx open-autonomy preflight; echo $?` — *Today:* no collision output whatsoever (only pty/lockfile lines). *After:* exit 1; output names **both** `termfleet` (self-reference: host package name is in the runner's dependency set) and `@termfleet/core` (workspace `packages/core` shadows the published dep), and includes at least one remediation plus the statement that npm cannot prefer the registry copy over a workspace link.
2. **Compile refuses loudly in the same repo.** `cd $REPO && npx open-autonomy compile simple-sdlc local . ; echo $?` — *Today:* compiles and prints `installed N files` (silent time bomb). *After:* exit 1 with the same named error before any file is written; `... compile simple-sdlc local . --force` still materializes (escape hatch).
3. **The emitted loop driver refuses before a tick.** After a `--force` compile into `$REPO` (or a compileLocal unit-test scaffold à la `scheduler-termfleet-guard.test.ts`): `node scheduler/run.mjs --once; echo $?` — *Today:* the guard at emit.ts:84 passes (`node_modules/termfleet` exists) and the run dies deep with `Cannot find module <repo>/dist/index.js`. *After:* exit 1 with the named collision error; stderr does **not** contain `ERR_MODULE_NOT_FOUND`. Encode as a new case in `packages/substrate-local/src/scheduler-termfleet-guard.test.ts` (run via `bun test packages/substrate-local/src/scheduler-termfleet-guard.test.ts`).
4. **Transitive collisions are caught with the owning chain.** In a variant fixture whose workspace member is named `ws` (a transitive dep of termfleet), `npx open-autonomy preflight` exits 1 naming `ws` and the dependency chain (e.g. `ws ← termfleet`). *Today:* silent.
5. **No false alarms.** In a clean non-colliding repo (`mktemp -d`; `npm init -y; npm install termfleet`), `npx open-autonomy preflight` prints the collision check's OK line and the check contributes no failure; `compile simple-sdlc local .` succeeds. A repo with workspaces whose names don't intersect the closure also passes. (Regression guard for the F-5 crying-wolf lesson.)
6. **Docs mention workspaces.** `grep -ci workspace docs/OPERATIONS.md docs/INSTALL-AGENT.md` — *Today:* 0 in the relevant install sections. *After:* ≥1 caveat in each, describing this exact failure class.
