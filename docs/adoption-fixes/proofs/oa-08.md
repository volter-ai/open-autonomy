# OA-08 proof — AC-7 ("PM launch-failed escalation") "live-pending" gap, closed

Spec: `docs/adoption-fixes/OA-08-launch-verification-and-dead-worker-escalation.md`. OA-08 landed
code-complete (AC-1..6, AC-8), but **AC-7** — the PM's 3-tick doctrine that escalates a repeatedly
failing launch (`launch-failed` -> one retry -> `human-required`, ending the tick with
`OUTCOME: blocked launch-failure <id>`) — was left **live-pending**: the spec's own AC-7 text says it
is "verified behaviorally, not unit-tested" (judgment-layer, model-driven). This proof closes the
**board-mechanics half** of that gap as a **committed, deterministic test**: a real compiled
`simple-sdlc` install, a real `ztrack` board, and a **canned PM driver** that replays the doctrine's
exact `ztrack` commands across 3 ticks over the REAL, already model-free runner-refusal path
(`runner-frontend.ts`'s pre-check, proven by OA-08's own `launch-verification.test.ts`) — zero model
calls, zero live process, and (the whole point) durable: it survives a box wipe because it lives in
git, not in `/workspace/proofs/`.

This file replaces an earlier, since-wiped `/workspace/proofs/oa-08.md` — the durability requirement
this proof exists to satisfy.

## 3. Per-tick board-state assertions, and how to reproduce

Fixture (`scaffold()` in `pm-escalation.test.ts`): a real `compileLocal(ir)` output (pm + develop
agents, `codeHost: 'github'` so a successful isolated launch would record a post-session effect
marker — making "no effect marker" a real discriminator, not vacuously true — mirrors
`launch-verification.test.ts`'s fixture IR exactly), written to a fresh `mkdtemp` dir, stub termfleet
installed (verbatim duplicate of `launch-verification.test.ts`'s `installStubTermfleet` — that file is
owned by another builder and was not imported or edited; this is this file's own copy per the task's
explicit instruction), a REAL `ztrack init --preset simple-sdlc` (the bundled preset this profile
declares via `policy.box.tracker.ztrackPreset`, `profiles/simple-sdlc/ir.yml`), a REAL
`ztrack issue create` (state `ready`, one pending dev AC so the board starts preset-valid modulo "not
done yet"), then `git init` + `git add -A` + `git commit` — a real git repo with a real committed
ztrack board. Then a COMMITTED, PERMANENT breakage: `git rm -r .claude/skills/develop && git commit`
(AC-1/AC-2's literal recipe; the skill is never restored across the 3 ticks — this is AC-7's own
"permanently broken develop skill" scenario).

Each tick calls `runCannedPmTick()` (`test-support/canned-pm-escalation.ts`), which:
- reads the issue's CURRENT labels via a REAL `ztrack issue view <id> --json` (never cached from a
  previous tick — a fresh read every time, matching "the board is the only memory across ticks");
- if `human-required` is already present: dispatches NOTHING (no subprocess spawned at all) — tick 3's
  behavior;
- otherwise: runs the REAL `bun scripts/runner.ts launch develop --ref <id> --branch agent/issue-<id>`
  (doctrine's exact command) and, based on its exit code and the labels read at the TOP of the tick,
  performs the exact `ztrack issue edit`/`comment` doctrine specifies (see the file's top comment for
  the full 1:1 mapping to SKILL.md :103-132).

**Tick 1** (`action === 'first-failure'`):
- `tick1.launch.status !== 0` — the REAL runner refused (`stderr` contains `launch refused`).
- `existsSync(sentinel) === false` — the stub `createAgentWindow` was never called: no session.
- `effectsCount(dir) === 0` — no post-session effect marker.
- `existsSync(worktree) === false` and `git rev-parse --verify agent/issue-<id>` fails — the
  just-created worktree+branch was torn down (OA-08's F1 recoverability guarantee, re-verified here
  independently of `launch-verification.test.ts`).
- `readLabels(dir, issueId) === ['launch-failed']` — an INDEPENDENT re-read off the board (not
  `tick1`'s own in-memory return value) — proves the label is durable, not just asserted from cached
  state.
- `readComments(dir, issueId)` contains the runner's `launch refused` error line.

**Tick 2** (`action === 'second-failure-escalate'`):
- Board re-read at the top of the tick (before any action) already shows `launch-failed` — the
  cross-tick memory working as designed.
- `tick2.launch.status !== 0` again (skill still missing — nothing fixed it).
- Same no-session / no-effect-marker / no-branch-or-worktree assertions as tick 1.
- `tick2.outcomeLine === 'OUTCOME: blocked launch-failure <id>'` — the exact doctrine string
  (spec :126, SKILL.md :126).
- `readLabels(dir, issueId)` now contains BOTH `launch-failed` and `human-required`.
- Comment count with `launch refused` is now 2 (one per failed tick, never more).

**Tick 3** (`action === 'skip-human-required'`):
- Board carries `human-required` at tick start.
- `tick3.dispatched === false` and `tick3.launch === null` — **no subprocess was even spawned**, i.e.
  no 3rd `launch develop` invocation happened at all (stronger than "it failed again" — it never ran).
- Label set unchanged (`tick3.labelsAfter` set-equal to the pre-tick-3 label set).
- `sentinel` still absent — zero sessions were ever created across all 3 ticks.
- Comment count with `launch refused` is still exactly 2 (not 3) — independent confirmation of "no 3rd
  invocation" from a completely different angle (board state, not the driver's return value).

Final assertion (the "not silently faked" check): a genuine `ztrack check <id> --json` runs against
the REAL preset and reports no finding other than the expected, still-open dev AC
(`ready_requires_dev_ac`/`dev_ac_incomplete`) — proving the label/comment writes themselves introduced
no preset-validation errors; the board stayed genuinely well-formed throughout, not merely
label-scribbled markdown.

## 4. HONEST BOUNDARY

This proves the **board-mechanics half** — `launch-failed` -> one retry -> `human-required` -> stop —
and the **REAL, model-free runner-refusal exit code**, deterministically and durably (it survives a
full repo wipe: nothing here depends on a live process, termfleet, or any external service; the only
non-builtin dependency is the repo's own pinned `ztrack@1.0.0` devDependency, installed by `bun
install` like any other).

It does **NOT** prove that a real-model PM, handed only `SKILL.md` and the board each fresh tick,
independently **reads** the doctrine and **decides** to take these exact actions. That judgment is
canned in `test-support/canned-pm-escalation.ts`'s `runCannedPmTick` — its function body is a direct,
literal transcription of SKILL.md's "## Failed launches" rule (:103-132), not independently derived by
a model. Model judgment over doctrine is inherently live-only and is **NOT** re-proven here — this is
consistent with:
- the spec's own AC-7 text ("Judgment-layer: verified behaviorally, not unit-tested — consistent with
  the repo's PM-doctrine testing posture");
- `CLAUDE.md`'s "Built vs designed" section, which already names the sibling case
  (escalate-on-SLA) as a live-proven `test.todo`, never a unit test.

**Named residual gap:** a live/bench run of the ACTUAL `pm` skill (model-driven, on a permanently
broken `develop` skill) reproducing this same 3-tick arc — dispatch, read the refusal, label
`launch-failed`, retry, label `human-required` + emit the `OUTCOME:` line, then dispatch nothing on
tick 3 — remains the only proof of the judgment layer itself. This file is deliberately **not** that
proof and does not claim to be. It closes the mechanical half of AC-7 (the half that doesn't need a
model to be true) as a committed, repeatable, durable test; the judgment half stays exactly where the
spec always said it belonged — live verification.
